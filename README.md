

TASK 1 CHECKLIST

1. Understand the Brief

☐ Read the brief at least twice
☐ Highlight key requirements
☐ Identify the target audience
☐ Identify the purpose of the solution
☐ List functional requirements (what it must do)
☐ List non-functional requirements (performance, accessibility, security etc.)
☐ Identify constraints (time, budget, technical limits)



2. Requirements & Planning

☐ Write a clear requirements summary
☐ Break the project into smaller tasks
☐ Create a basic project timeline
☐ Identify potential risks + mitigation
☐ Create user stories or use cases (if required)
☐ Decide what technologies you will use (HTML/CSS/JS/SQL etc.)

---

3. Design Stage

☐ Create wireframes (mobile + desktop if required)
☐ Show navigation structure
☐ Plan layout clearly
☐ Choose colour scheme (justify why)
☐ Choose typography (justify why)
☐ Consider accessibility:

* ☐ Colour contrast
* ☐ Alt text
* ☐ Readable fonts
* ☐ Keyboard navigation

☐ Justify ALL design decisions

---

💻 4. Development Stage

### Structure

☐ Semantic HTML
☐ Clean file structure
☐ External CSS file
☐ Organised folders

### Functionality

☐ Navigation works
☐ Forms work
☐ Input validation
☐ Error messages
☐ Interactive features function correctly

### Code Quality

☐ Indentation correct
☐ Meaningful variable names
☐ Comments added where needed
☐ No unused code

### Security (if relevant)

☐ Basic input sanitisation
☐ No exposed sensitive data

---

## 🧪 5. Testing

☐ Create a test plan table
☐ Test each functional requirement
☐ Record expected vs actual result
☐ Fix issues found
☐ Re-test after fixes

Test:
☐ Different screen sizes
☐ Different browsers
☐ Invalid form inputs
☐ Edge cases

---

## 📊 6. Evaluation

☐ Compare final product to original requirements
☐ Explain what works well
☐ Identify limitations
☐ Suggest improvements
☐ Reflect on development process

---

# ⭐ What Gets You Higher Marks

To push into distinction-level work:

* Explain WHY you made decisions
* Link everything back to the client brief
* Use professional terminology
* Show logical structure in documentation
* Keep everything consistent

Markers love:

> “This decision meets requirement 2.1 because…”
